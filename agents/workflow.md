# Potex Operating Workflow

## 목표
DevZero의 기존 운영방식에서 효과적인 부분만 흡수해서 Potex에 맞게 최적화한다.

핵심은 아래 4가지다:
- 세션 복구 가능성 유지
- 역할 분리
- 작업 범위 명확화
- 결과 검증 후 보고

## 역할 구조
### Hermes
- 전체 오케스트레이션
- 현재 상태 파악
- 작업 분해
- 문서/운영/구현 중 어떤 성격인지 판단
- 최종 검토 및 사용자 보고

### Worker agents (필요 시)
- 구현성 작업 수행
- 코드/스크립트 수정
- 반복 검증
- 변경 범위 보고

## Potex에 맞춘 최적화 원칙
DevZero의 원형을 그대로 복제하지 않고, Potex에는 아래처럼 경량 적용한다.

### 유지할 것
- 프로젝트별 폴더 격리
- 세션 체크포인트
- 재개 프로토콜
- 작업 단위 분리
- 완료 전에 검증

### 단순화할 것
- 복잡한 브리지/워처는 꼭 필요할 때만 도입
- 초기에는 `agents/session.md` 중심으로 상태 관리
- 문서 작업과 데이터 운영 작업은 같은 프로젝트 폴더 안에서 추적

## 작업 시작 순서
1. `agents/session.md` 확인
2. 현재 요청이 문서 정리 / 데이터 운영 / 구현 작업 중 무엇인지 분류
3. 관련 핵심 문서만 읽기
4. 직접 처리 가능한 범위인지 판단
5. 구현성·대량 작업이면 워커 위임 범위를 먼저 확정
6. 결과를 검증한 뒤 session checkpoint 갱신

## 문서 배치 원칙
- 프로젝트 개요: `README.md`
- 세션 상태: `agents/session.md`
- 운영 방식: `agents/workflow.md`
- 도메인 운영 문서: 루트 `.md` 문서들
- 코드: `potex-gas/`
- 산출물: `generated/`, `outputs/`, `workdata/`

## 데이터 운영 원칙
- 원본 Google Sheets는 읽기 전용 소스로 취급
- 정규화와 운영 판단은 canonical/ops 레이어 기준으로 관리
- 이름 정규화는 가능한 풀네임 우선
- assigned coach 필드에 나온 이름은 provisional coach로 보존 가능
- 비기술 운영자를 위해 spreadsheet-first 흐름 유지

## 완료 기준
작업이 끝났다고 보려면 아래를 만족해야 한다.
- 관련 파일/문서가 실제로 반영됨
- 필요한 검증이 끝남
- session checkpoint가 최신 상태로 갱신됨
- 다음 재개자가 바로 이어받을 수 있음
