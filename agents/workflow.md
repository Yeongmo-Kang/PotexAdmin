# Potex Operating Workflow

## 目的
DevZero の既存運用で有効だった部分だけを取り込み、Potex 向けに軽く最適化する。

大事なのは次の 4 点。
- セッションを途中から再開できること
- 役割を分けること
- 作業範囲を明確にすること
- 結果を検証してから報告すること

## 役割構造
### Hermes
- 全体の進行管理
- 現在状態の把握
- 作業の分解
- 文書 / 運用 / 実装のどれかを判断
- 最終確認とユーザー報告

### Worker agents（必要時のみ）
- 実装系の作業実行
- コード / スクリプト修正
- 繰り返し検証
- 変更範囲の報告

## Potex 向けの最適化原則
DevZero の原型をそのまま複製せず、Potex では次の形で軽量運用する。

### 維持すること
- プロジェクトごとのフォルダ分離
- セッションチェックポイント
- 再開プロトコル
- 作業単位の分離
- 完了前の検証

### 簡略化すること
- 複雑な bridge / watcher は本当に必要な時だけ導入する
- 初期運用は `agents/session.md` を中心に状態管理する
- 文書作業とデータ運用作業は同じプロジェクトフォルダ内で追跡する

## 作業開始の順番
1. `agents/session.md` を確認する
2. 今回の依頼が文書整理 / データ運用 / 実装作業のどれかを分ける
3. 関連する核心文書だけを読む
4. 直接処理できる範囲かを判断する
5. 実装系や大量作業なら、先に worker へ渡す範囲を決める
6. 結果を検証してから session checkpoint を更新する

## 文書配置の原則
- プロジェクト概要: `README.md`
- セッション状態: `agents/session.md`
- 運用方式: `agents/workflow.md`
- 業務運用文書: ルートの `.md` 文書
- コード: `potex-gas/`
- 生成物: `generated/`, `outputs/`, `workdata/`

## データ運用の原則
- 元の Google Sheets は read-only source として扱う
- 正規化と運用判断は canonical / ops レイヤー基準で管理する
- 名前の正規化は、できるだけフルネーム優先で行う
- assigned coach フィールドの名前は provisional coach として保持してよい
- 非技術 operator のため、spreadsheet-first の流れを維持する

## 完了条件
作業完了とみなすには、次を満たすこと。
- 関連ファイル / 文書に実際の変更が反映されている
- 必要な検証が終わっている
- session checkpoint が最新に更新されている
- 次の担当者がそのまま再開できる
