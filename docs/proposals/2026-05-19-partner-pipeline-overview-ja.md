# POTEX データ一元化と各チームの業務効率化 ― 現状と今後の追加（全社共有）

> POTEXでは、「データの本体は1か所にまとめて、各チームはそれぞれの視点でそれを見る」 という方針で仕組みを整えてきています。
> 本書は、その全体像（実装済み・実装予定の両方）と、どの業務が自動化・効率化されるかを、全社で共有することを目的としています。
> 細かい運用ルールの確認事項は、後日まとめて別途お送りします。

---

## 1. 全体方針

- **データの本体は1か所だけ** に集めます（`POTEX DB`）。
- 各チームは、その本体を **自分の業務に必要な切り口で見る** 専用ワークブックを持ちます。
- 同じ事実を Slack・Lステップ・スプレッドシート・記憶に分散させず、本体を1か所にまとめることで、**各チームの業務を自動化・効率化する** ことを狙っています。

## 4. 仕組みによって自動化される処理

### 4-1. すでに自動化が動いている部分（`POTEX DB` 内部）

- 名前のゆれの自動吸収（例：`坂本` ↔ `坂本直樹`、`稲川コーチ` ↔ `稲川 雄介`）
- 顧客と支払いの自動紐付け／紐付かないものは `Potex CS` の確認キューへ自動投入
- 顧客と継続契約の自動紐付け／紐付かないものも同様に確認キューへ
- フィードバック点数と顧客の自動マッチング
- LINE登録の流入元（YouTube／インフルエンサー／インスタ／TikTok など）の自動分類と集計
- 顧客ライフサイクル（LINE登録 → リード → 体験予約 → 契約 → 入金 → 完了／失注）の自動追跡
- 各種健全性指標（件数・未マッチ・流入分布 など）の自動集計
- データ更新：入力収集 30分ごと／再表示 1時間ごと／全体更新 毎日朝7時

### 4-2. 実装予定の自動化（既存4ワークブック運用開始＋実装予定3ビュー）

- **CS確認キュー → 1クリック承認 → 本体反映**：今までPython script等で行っていた名前マッチ決定が、`Potex CS` 上で1行承認するだけで本体に反映される
- **partner担当leadの一覧化**：partner本人が状況入力するだけで、全社が同じ状況を同時に見られる
- **30日更新なしの自動検出**：長くフォロー漏れになっている顧客／lead が一目で分かる
- **営業の契約・入金状況の自動可視化**：Slack遡りなしで、自分のシートで一覧確認できる
- **コーチの担当負荷・要フォロー自動アラート**：誰に何件溜まっているか、コミュニケーション履歴を遡らず把握できる

## 5. 各ワークブックで見えるもの

### 5-1. `Potex CS`（実装完了）
- フォローアップ対象顧客の一覧

- 健全性指標（未マッチ件数、未承認件数、フォロー漏れ件数）
- 流入元分布
- 期間別変動

### 5-3. `Potex Concierge`（実装完了）
- Concierge担当のフォロー対象一覧
- 担当領域の健全性サマリ

### 5-4. `Potex Sales`（実装完了）

- `営業_データ状況`：健全性指標（未マッチ支払い39件、入金済51件、契約イベント150件 等）

### 5-5. `Potex Coaches`（実装完了）
- `コーチ_使い方`：ビュー説明

- `コーチ_要フォロー一覧`：要フォローアラート（フィードバックや満足度起点、18件）
- `コーチ_データ状況`：健全性指標（コーチ数38、要フォロー顧客延べ、低満足度件数 等）

### 5-6. 稲井さん・佐藤さん用ビュー（実装予定）

partnerごとに同じ4タブ構成。詳細仕様は別途お送りしますが、概要は以下のとおりです。

- **アサイン一覧タブ**（自動表示）：今割り当てられているlead が1人1行で見える。表示項目は アサイン日／表示名／面談状況／POTEX商品状況／人材側状況（インターン・選考・成約 など）／状態メモ／最終更新日。
- **状況入力タブ**（partner本人が入力）：lead 1件ずつ、面談結果・POTEX商品・人材側ステータス・状態メモを入力して「送信する」にチェック。30分以内に `POTEX DB` に自動反映され、`Potex CS` や `Potex Executive` 側にも同時に反映される。
- **アサイン操作**：partner側では行わず、引き続きCSが `Potex CS` の「partner割り当て入力」タブで行う。
- **使い方タブ／ヘルスチェックタブ**：入力ルール案内と件数サマリ。

これにより、**今までSlackやLステップに文章でしか残っていなかった partner pipeline の状態が、初めて構造化されたデータとして仕組みに乗る** ようになります。

## 6. 各チームの業務がどう変わるか（運用開始・実装後）

| チーム | 今までの状況 | 仕組み運用後 |
|---|---|---|
| **CS** | Form / カレンダー / Lステップ / Slack を行き来して状況把握、名前マッチ決定はPython等 | `Potex CS` 上で「対応待ち」「未マッチ」「partner担当lead最新状況」が一覧。名前マッチ承認も `Potex CS` 上で1クリック |
| **営業** | Slack で結果報告、契約・入金状況は個別管理 | Slack報告は今までどおり。営業用ビュー実装後は、契約状況・入金待ち・ファネルを自分のシートで直接確認できる |
| **コーチ** | 担当顧客の状況は記憶＋コミュニケーション履歴 | コーチ用ビュー実装後は、自分の担当負荷・要フォロー対象がシートで自動表示 |
| **稲井さん／佐藤さん** | 担当lead の状況は記憶＋Lステップ＋Slackの混在 | 自分のシートに担当leadが一覧。面談後に状況を入れるだけで全社に共有 |
| **経営** | `Potex Executive` ＋ ヒアリング | Executive側にpartner進捗も自動で乗る。フォロー漏れも自動検出 |
| **Concierge** | 既存 `Potex Concierge` | 今回の追加領域とは独立。既存どおり |

## 7. 既存運用との関係（変わらないもの）

新しい仕組みは、**今の運用を置き換えるものではなく、上に乗せるもの** です。

- YouTube・LINE登録：引き続き Lステップが入口
- Form予約：引き続き Form が入口
- アサイン：引き続きCSの判断で行う。入力場所が1つ増えるだけ
- 面談：面談中の運用は変わらない
- 結果報告：従来の「営業 → Slack → CSが Lステップに記載」のフローは続けて構わない

---

## 8. 全体としての期待効果

- 1か所のデータを各チームが**それぞれの視点で同時に見られる**
- CS：Slack や Lステップを遡らず、必要な情報が `Potex CS` で揃う／名前マッチ決定が1クリック化
- 経営：全社進捗を `Potex Executive` で直接確認、フォロー漏れも自動検出
- 営業／コーチ／partner：それぞれ自分の担当領域を直接確認できる
- 「最終更新日」が自動で残るため、長く動きのない件が一目で分かる
- 将来、新しいチームや担当領域が増えても、**同じ仕組みをそのまま使い回せる**
